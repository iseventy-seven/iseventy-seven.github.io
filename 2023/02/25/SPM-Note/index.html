A2 知识体系

A21 三个基本概念

  • 项目（Project）

    • 项目定义（Project Definition）

      • A Project is a temporary endeavor undertaken to create a unique product, service, or result.

        • temporary

          • 有明确的起点和终点
            • 项目结束条件
              • 达成目标
              • 无法达成目标
              • 资金短缺
              • 需求不复存在
              • 人力物力缺乏
              • 法律等原因被迫中止
          • 持续时间或长或短
          • 项目临时，成果长存

        • product

          • 独特的产品，或其他产品的组成部分、某个产品的升级版和修正版。

        • service

          • 独特的服务或提供某种服务的能力。

        • result

          • 独特的成果。

      • A project is a sequence of unique, complex, and connected **activities **that have one goal or purpose and that must be completed by a specific time, within budget, and according to specification.

        • activities 活动序列
        • goal or purpose 目标
        • time 时间
        • budget 预算
        • specification 规约

    • 项目属性（Project Attributes）

      • Unique Purpose 特定目标
      • Temporary 临时性的
      • Drives Change and Enable Value Creation 推动变革，创造价值
      • Progressive Elaboration 逐步精化
      • Requires Resources 需要资源
      • Customer or Sponser 有发起人
      • Uncertainty 不确定性
      • Managers 有经理人

    • 项目管理（Project Management）

      • 在项目活动中运用专门的知识、技能、工具和技术，以满足项目需求。
      • 软件项目管理
        • 为了满足软件项目需求，软件项目经理管控整个软件开发过程，最终使项目获得成功的知识、技能、工具和技术。
        • 识别需求 + 管理期望 + 权衡关系 + 滚动规划

  • 项目群（Programs）

    • 一组相互联系的项目，宜使用协同方法进行管理来获得收益和进行控制，而这种收益和控制在单独管理这些项目时是不易获得的。

  • 项目组合（Project Portfolio）

    • 组织将项目以及项目群组合并进行管理，使其作为一个投资组合，从而促成整个企业的成功。

  • 点 — 项目管理 — Tactical Goals 战术目标 -> 线 — 项目群管理 — Campaign Goals 战役目标 -> 面 — 项目组合管理 — Strategic Goals 战略目标

B2 过程模型

B21 PMLC 模型

  • PMLC

    • Project Management Life Cycle

  • 模型分为三层五类

  • 传统模型

    • 特征

      • 低复杂性
      • 范围变更请求很少
      • 技术架构很好理解
      • 低风险
      • 适于有经验、有能力的开发团队
      • 计划驱动的 TPM 项目

    • 具体类型

      • The Linear PMLC model 线性模型

        • 优点
          • 开始时就进行进度安排，且资源已知，较程式化。
          • 不需要熟练的团队或同地协作，风险低。
        • 缺点
          • 不适应变更
          • 成本过高
          • 耗时过长
          • 需要完整、细致的计划和严格的过程组线性顺序
          • 过程中不关注客户价值
        • 种类
          • The Standard Waterfall model 标准瀑布模型
          • Rapid Development Waterfall model 快速开发瀑布模型
          • Rapid Development Waterfall model Ⅱ 快速开发瀑布模型 Ⅱ
            • ![24.快速开发瀑布模型 Ⅱ](........\pictures\SPM\24.快速开发瀑布模型 Ⅱ.png)

      • The Incremental PMLC model 增量模型

        • 优点
          • 更早产生商业价值
          • 更好安排稀缺资源
          • 适应小变更
          • 产品有改良机会
          • 比线性模型更加关注客户价值
        • 缺点
          • 团队人员不稳定
          • 增量之间需要大量交接文档
          • 增量优先级需要考虑依赖性
          • 更需要客户参与
          • 比线性模型耗时更长
          • 可能有功能分割问题
        • 种类
          • The Incremental PMLC model 增量项目管理生命周期模型
          • The Incremental PMLC model Ⅱ 增量项目管理生命周期模型 Ⅱ
            • ![27.增量项目管理生命周期模型 Ⅱ](........\pictures\SPM\27.增量项目管理生命周期模型 Ⅱ.png)
          • Staged Delivery Waterfall model 分阶段交付瀑布模型
          • Feature-Driven Development model 特性驱动的开发模型

  • 敏捷模型

    • 特征

      • 没有已知的解决方案
      • 预计会有新的商业机会
      • 变化驱动的 APM 项目
      • 化解风险非常关键
      • 客户参与其中
      • 使用小型同地协作团队

    • 具体类型

      • Iterative PMLC model 迭代模型

        • 优点
          • 可变更范围
          • 适应变化
          • 关注形成商业价值
          • 客户可评审部分解决方案
        • 缺点
          • 团队人员不稳定
          • 需要同地协作
          • 资源需求不清晰
          • 客户参与度需求更高
          • 最终解决方案不确定
        • 种类
          • Prototyping model 原型模型（丢弃原则，重新做原型）
          • Evolutionary Development Waterfall model 进化开发瀑布模型（进化原则，不断修改原型）
          • Dynamic Systems Development Method 动态系统设计方法
          • Rational Unified Process model RUP 模型（统一过程模型）
            • ![34.RUP 模型](........\pictures\SPM\34.RUP 模型.png)
          • The Scrum process flow 敏捷开发模型流程

      • Adaptive Software Development model 软件开发适应性模型

        • 优点
          • 不断重组项目管理过程以适应变化
          • 避开变更请求中的管理工作
          • 不在不确定的规划上浪费时间
          • 以商业价值为导向的演进
        • 缺点
          • 需要客户的有效参与
          • 不能识别最终成果

  • 极限模型

    • 特征

      • xPM 通常是研发项目
      • xPM 项目的风险很高

    • 具体类型

      • Extreme PMLC model 极限模型

        • 优点

          • 备选方案保持时间长
          • 能提供大量部分解决方案供早期参考

        • 缺点

          • 可能在所有错误的地方寻找解决方案
          • 不能保证项目交付结果具有商业价值

        • 种类

          • The INSPIRE Extreme PMLC model INSPIRE 极限项目管理生命周期模型

  • 对比

    • 相同点
      • 都包含五过程组
      • 都始于范围/启动过程组
      • 都终于收尾过程组
    • 不同点（自上而下）
      • 不确定性越来越高
      • 风险管理越来越重要
      • 客户有效参与需求越来越高
      • 趋向生命周期的开端（反馈趋前）
      • 即时项目计划取代完整项目计划

B23 敏捷模型

  • 敏捷宣言
    • 个体和互动高于流程和工具
    • 工作的软件高于详尽的文档
    • 客户合作高于合同谈判
    • 响应变化高于遵循计划
  • 12 条原则
    • 敏速
      • 尽早交付：通过尽早和持续地交付有价值的软件来满足客户。
      • 不断交付：要不断交付可用的软件，周期从几周到几个月不等，且越短越好。
      • 以用为尺：可用的软件是衡量进度的主要指标。
    • 敏变
      • 拥抱变化：欢迎对需求提出变更，即使是在项目开发后期。要善于利用需求变更，帮助客户获得竞争优势。
      • 精益求精：对技术的精益求精以及对设计的不断完善将提升敏捷性。
      • 大道至简：要做到简洁，即尽最大可能减少不必要的工作。
    • 敏合
      • 并肩作战：项目过程中，业务人员与开发人员必须在一起工作。
      • 保持稳定：提倡可持续的开发。项目方、开发人员和用户应该能够保持恒久稳定的进展速度。
      • 对面交谈：无论是团队内还是团队间，最有效的沟通方法是面对面的交谈。
    • 敏我
      • 自我组织：最佳的架构、需求和设计出自于自组织的团队。
      • 自我改进：团队要定期反省如何能够做到更有效，并相应地调整团队的行为。
      • 善于激励：要善于激励项目人员，给他们以所需要的环境和支持，并相信他们能够完成任务。

C22 WBS

  • 基本概念

    • 通过 WBS 工作分解结构加强项目范围管理。
    • 定义
      • WBS是把项目细分为更小的、更易于管理的组成部分的一个层级结构。每向下一个层级，就是对项目工作的更详细的定义。
      • WBS是对项目团队为实现项目目标、创建可交付成果而需要实施的全部工作范围的层级分解。WBS组织并定义了项目的总范围。
    • 概念辨析
      • WBS 是对项目工作的分解或划分，以便更好的界定、沟通和管理项目工作范围。
      • WBS 向干系人展示了即将开展的项目工作的全景图。
      • WBS 是一种分而治之的方法。
      • WBS 不是我们必须做什么任务，而是我们必须交付什么。
      • WBS 不是该怎样或何时交付，而是要交付什么。
      • WBS 不是活动清单，不是进度安排，而是范围工具。

  • 优点

    • 开发者 产品经理 项目经理 干系人
      尽早理解范围 防止变更失控 分配解释工作 改善沟通环境
      交付期望成果 提供变更基准 打下采购基础 促进共识达成
      理解陌生领域 促进产品规划 提高汇报水平 获得涉众认同
      区分内外工作 尽早发现问题 监控项目工作 建立信赖基础
      明确项目边界 发现重叠工作 提高管理水平
      整合时间成本

  • 结构要素

    • 层级

      • 从第一层级开始

    • 标志及编码

    • 组件和要素

      • WBS 组件
        • 是 WBS 中任何层级上的条目。
        • 所有组件联合决定了项目的总范围。
      • WBS要素
        • WBS 中任一 WBS 组件及其 WBS 属性的综合。
        • WBS 要素 = WBS 组件 + 属性。

    • WBS 组件类型

      • Discrete 离散的
        • 分立型
      • Level of Effort 努力程度
        • 人力投入型
        • 支持型（活动）

    • 工作包

      • 位于 WBS 层级结构底层的 WBS 组件。

    • 100% 规则

      • WBS 涵盖了项目范围中的 100% 的工作（包括项目管理），列出了将要完成的全部可交付成果。
      • 上层的某个组件必须正好等于下层所有组件的工作之和。

    • 命名

      • 名词和形容词，不能用动词。
      • 清晰的反映工作范围。

  • 基本概念

    • 项目进度管理包括为管理项目按时完成所需的各个过程。
    • 项目过程中使用一些基本的项目管理工具和技术，就能够改善项目的进度管理。
    • 通过制定详尽计划，说明项目如何及何时交付项目范围中定义的产品、服务和成果，并作为一种用于沟通和管理相关方期望的工具，以及报告绩效的基础。

        • 资源平衡

          • 定义
            • 为了在资源需求与资源供给之间取得平衡，根据资源制约因素对开始日期和完成日期进行调整的一种技术。
          • 场景
            • 共享资源或关键资源只在特定时间可用，数量有限。
            • 被过度分配，如一个资源在同一时段内被分配至多个活动。
            • 为保持资源使用量处于均衡水平。
          • 后果
            • 资源平衡往往导致关键路径改变。
          • 方法
            • 可以用浮动时间平衡资源。

        • 进度压缩技术

          • 定义
            • 在不缩减项目范围的前提下，缩短或加快进度工期，以满足进度制约因素、强制日期或其他进度目标。
          • 方法 1 — 赶工
            • 通过增加资源，以最小的成本代价来压缩进度工期的一种技术。例如批准加班、增加资源、支付加急费用等。
          • 方法 2 — 快速跟进
            • 一种进度压缩技术，将正常情况下按顺序进行的活动或阶段改为至少是部分并行开展。例如，在大楼的建筑图纸尚未全部完成前就开始建地基。

C32 关键路径

  • 基本概念

    • 关键路径法 Critical Path Method, CPM
    • 是一种网络图技术，用来预测整个工期。
    • 是决定项目最早完成时间的活动序列。
    • 是网络图的最长路径，其时差或者浮动时间最少。
    • The fact that its name includes the word critical does not mean that it includes all critical activities. 关键路径仅仅与项目的时间维有关，它的名字中包含”关键”并不意味着它包含所有的关键活动。
    • There can be more than one critical path if the lengths of two or more paths are
      the same. 一个项目可以有多条关键路径。（最少一条）

  • 关键路径图

    • 最早开始时间 ES Early Start
      • 一个活动的最早开始时间是基于项目网络逻辑可以开始的最早的可能时间。
      • 等于前置活动最早结束时间中的最晚时间。
    • 最早完成时间 EF Early Finish
      • 一个活动的最早完成时间是基于项目网络逻辑最早可能完成的时间。
      • 等于最早开始时间加当前活动持续时间（Duration）。
    • 最晚开始时间 LS Late Start
      • 一个活动的最晚开始时间是一个项目在不延迟项目完成时间的最晚可能开始的时间。
      • 对于关键路径，最早开始时间和最晚开始时间相同；对于非关键路径，二者不相同。
      • 等于最晚完成时间减当前活动持续时间。
    • 最晚完成时间 LF Late Finish
      • 一个活动的最晚完成时间是一个活动在不延迟项目完成时间的最晚可能完成的时间。
      • 等于最晚开始时间加当前活动持续时间。
      • 等于后置活动最早开始时间中的最早时间。
    • 时差/浮动时间
      • 在不延迟后续活动或项目完成日期的情况下，活动可以延迟的时间量。
      • 总时差/总浮动时间 TS Total Slack/Total Slack
        • 一个活动从它最早开始时间起，在没有拖延计划项目完成日期的情况下可以被耽搁的时间。
        • 等于最晚开始/结束时间减最早开始/结束时间。
        • 关键路径上的总时差都为零。
      • 自由时差/自由浮动时间 FS Free Slack/Free Float
        • 一个活动在不延误紧接活动的最早开始时间的情况下可以被延误的时间。
        • 等于紧后活动的最早开始时间减当前活动的最早结束时间。
    • 推导方法
      • Forward pass（正推法）
        • Determines the early start and finish dates. 推导最早开始时间和最早结束时间。
      • Backward pass（逆推法）
        • Determines the late start and finish dates. 推导最晚开始时间和最晚结束时间。
      • 特例
        • 一般情况下，最后一个结点的最早开始时间和最晚开始时间相同。

C42 功能点评估

  • 功能点评估

    • 评估标准

      • 标准 功能点类型 应用范围 操作难度 现状
        IFPUG 标准 五种功能点：ILF、EIF、EI、EO、EQ。 适用于所有项目，尤其适用于 MIS 类项目。 适中。
        IFPUG 功能点标准基于明确的规则约束，因而能保证软件规模评估结果的客观性和一致性。因为 IFPUG 是目前应用最普遍的功能点度量方法，因而有较多的资源可以借鉴和参考，例如 IFPUG 功能点相关的书籍、标准、文章、论坛、从业人员等资源都可以方便地得到。        		 应用最广泛的一种标准。
        MarkII 标准 一种功能点：逻辑事务。 适用于所有项目，尤其适用于 MIS 类项目。 简单。
        MarkII 功能点标准操作简单，只需进行简单的加权计算即可。但标准缺乏对基本元素的识别规则，例如对数据元素、逻辑事务仅采用举例的方式加以说明，实际操作过程中可能会出现歧义，度量结果的一致性不强。        		 主要在英国使用。
        NESMA 标准 五种功能
        点：ILF、
        EIF、EI、
        EO、EQ。        		 适用于所有项目，尤其适用于 MIS 类项目。 适中。
        NESMA 标准对 IFPUG 标准作了最“忠实”的借鉴，但也作了一些有别于 IFPUG 的规定，例如对代码数据进行度量、对隐含查询不予考虑，这些规定尽管在实际操作中有其合理的一面，但是却破坏了 IPFUG 规则的完整性和一致性。        		 主要在荷兰使用。
        COSMIC 标准 四种功能类型：进入、离开、读、写。 适用于所有项目，尤其适用于实时类应用。 较高。
        COSMIC 借鉴了 IFPUG 功能点标准中基于规则约束的实践，但引入了分层（Layer）概念，在评估功能规模时需要考虑软件的技术因素，从而减弱了软件功能点对业务人员透明的优势。        		 主要在加和欧部分国家使用。
        FISMA 标准 七种功能类型 适用于所有项目，要求了解被度量应用的技术实现方式。 很高。
        FISMA 强调采用“服务”替代“功能”，因而更能反映软件的特性。但其功能分类的数量繁多，并且每种服务都采用不同数值的加权系数，因而操作很困难。        		 主要在芬兰使用。

    • IFPUG

      • 五类功能点
        • 内部逻辑文件 ILF Internal Logical Files
          • 内部逻辑文件是一组可由用户识别确认的、在应用系统边界内维护的、逻辑上相关的数据或控制信息。
          • ILF 的主要意图是通过被测应用程序的一个或多个基本处理来保存维护数据。
        • 外部接口文件 EIF External Interface Files
          • 外部接口文件是一组可由用户识别确认的、由被测应用系统引用，但是在其它应用系统边界内维护的、逻辑上相关的数据或控制信息。
          • EIF 的主要意图是存放被测应用程序中的一个或多个基本处理所引用的数据。也就是说，一个应用程序的 EIF 必定是另一个应用程序的 ILF。
        • 外部输入 EI External Inputs
          • 外部输入是处理来自应用程序边界外的数据或者控制信息的一个基本处理，如用户通过增、删、改来维护内部逻辑文件。经过处理的数据维护一个或多个 ILF，经过处理的控制信息则可能维护一个 ILF 或不维护它。
          • EI 的主要意图是维护一个或多个 ILF，以及（或者）通过其处理逻辑来改变应用程序的行为。
        • 外部输出 EO External Outputs
          • 外部输出是一个向应用程序边界外提供数据或控制信息的基本处理。
          • EO 的主要意图是向用户提供经过处理逻辑加工的、除了检索数据或控制信息之外的信息或附加信息。处理逻辑必须包含至少一个数学公式或计算、创建导出数据、维护一个或多个 ILF，并且／或者改变系统的行为。
        • 外部查询 EQ External Inquiries
          • 外部查询是应用程序向其边界之外提供数据或控制信息的基本处理。
          • EQ 的主要意图是向用户展示未经处理、直接查询的一些数据或控制信息。处理逻辑不包含数
            学公式或计算，也不产生派生的数据。处理过程中外部查询既不维护 ILF，也不改变系统的行为。
